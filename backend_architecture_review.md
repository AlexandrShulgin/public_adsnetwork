Этот документ описывает ключевые архитектурные решения, принятые для обеспечения высокой производительности и масштабируемости платформы.

## 1. Минимизация Latency (Target: <50ms)

Основной вызов в AdTech — максимально быстрая подача объявления. Мы достигаем этого через:
- **Edge Computing**: API роуты работают на уровне Edge (Cloudflare Workers), максимально близко к пользователю.
- **In-Memory Auction**: Использование **Durable Objects** позволяет держать данные об активных кампаниях прямо в оперативной памяти воркера, избегая запросов к классической БД во время аукциона.
- **Distributed Indexing**: Использование Cloudflare KV для хранения индексов таргетинга (GEO, DeviceType).

## 2. Обработка данных (Write Path)

Для предотвращения деградации производительности БД при высоком трафике используется асинхронный пайплайн:
1.  **Ingestion**: Роут `/api/track` принимает событие и мгновенно помещает его в очередь (Cloudflare Queue).
2.  **Buffering**: Очередь сглаживает пики нагрузки.
3.  **Consumption**: Воркер [queue-consumer.ts](file:///c:/Users/sasha/Desktop/adsplatform/src/worker/queue-consumer.ts) вычитывает пачки событий и выполняет пакетную запись (Batch Insert) в PostgreSQL (Supabase), минимизируя количество транзакций.

## 3. Контроль бюджетов в реальном времени

Проблема "Double Spending" решается через:
- Централизованный бюджетный кеш в KV.
- Синхронизацию Durable Objects с KV для мгновенной приостановки аукциона по кампании, если ее баланс исчерпан.

## 4. Безопасность и Anti-Fraud

- **Bot Management**: Проверка `cf-bot-score` на уровне Edge позволяет блокировать подозрительный трафик до того, как он создаст нагрузку на систему аукциона.
- **Deduplication**: Использование временных ключей в KV для предотвращения повторного учета одного и того же клика/показа в течение короткого окна (Idempotency).
